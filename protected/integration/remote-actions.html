1. Best Practices

We’re flexible on how remote action endpoints are structured — we’ve integrated REST, GraphQL, and various custom APIs across different CPMS providers. There is no rigid format you need to follow. That said, here are patterns that work well based on what we see across the industry:

PracticeDetails
Standard OCPP actions The core set we typically integrate is reset (usually only hard), stop transaction, unlock connector, and occasionally remote start. These map directly to standard OCPP operations. ~80% of providers we integrate with support all three core actions.
Custom actions For non-OCPP actions (e.g. opening a station barrier), we handle these the same way — as long as they follow a similar request/response pattern, we can integrate any custom action.
ID granularity Some actions are station-level (e.g. reset), others connector-level (e.g. unlock, stop). This doesn’t need to be standardised — it makes sense for the ID to represent what the action operates on. See Section 4.
Synchronous confirmation
(~5 seconds)		 The endpoint should confirm whether the command was accepted or rejected within ~5 seconds. It doesn’t need to confirm the charger has completed the action — just that the CPMS has accepted it. This is the dominant pattern across all our integrations.
Clear error reasons A human-readable reason when rejected (e.g. "charger offline", "no active session") lets us relay useful information to the end user immediately. Error codes work too, but human-readable messages are easier to process in real-time conversation.
Stop charging
(special case)		 Many providers require a two-step process: query active sessions for the transaction ID, then stop that transaction. The simplest approach is a stop command directly at connector level (“stop whatever is charging on this connector”). About half our providers support this natively.
Response structure A simple status (accepted / rejected / failed) plus a message field. Standard HTTP codes (200, 400, 404, 429) for the rest.
Authentication We support OAuth2, API keys, and custom header-based auth — whatever you already use. See Section 5.

2. Typical API Structure

Below is a representative endpoint structure based on the most common patterns we encounter. Your existing API may differ — that’s fine. This is intended as a reference, not a specification.

ActionTypical EndpointID LevelNotes
Reset (hard) POST /api/v1/chargers/{id}/reset Station Most providers only expose hard reset. Soft reset is a nice-to-have.
Stop transaction POST /api/v1/connectors/{id}/stop Connector (preferred) or Session Connector-level is simplest. If session-level, we need an active-sessions endpoint.
Unlock connector POST /api/v1/connectors/{id}/unlock Connector Always connector-level — targets a specific plug.
Remote start POST /api/v1/connectors/{id}/start Connector Optional. Only some providers expose this.
Flexibility: We have providers using path-based IDs (/chargers/123/reset), query parameters (?serialNumber=ABC), request body IDs, and even GraphQL mutations. All work fine — the key is clear documentation of which identifier each action expects.

3. Request & Response Examples

3a. Reset

Request

POST /api/v1/chargers/STATION-1902/reset
Authorization: Bearer {token}
Content-Type: application/json

{
  "resetType": "HARD"
}

Success response 200 OK

{
  "status": "accepted",
  "message": "Reset command sent to charger"
}

Error response 400 Bad Request

{
  "status": "rejected",
  "message": "Charger is offline"
}

3b. Stop Transaction

Option A — Connector-level (preferred)

POST /api/v1/connectors/EVSE-8171-2/stop
Authorization: Bearer {token}

# Response 200 OK:
{
  "status": "accepted",
  "message": "Stop command sent to connector"
}

Option B — Session-level (also common)

# Step 1: Get active session
GET /api/v1/connectors/EVSE-8171-2/active-session

# Response:
{
  "sessionId": "TXN-90421",
  "connectorId": "EVSE-8171-2",
  "startTime": "2026-04-07T08:12:00Z",
  "energyDelivered": 14.3
}

# Step 2: Stop the session
POST /api/v1/sessions/TXN-90421/stop

# Response 200 OK:
{
  "status": "accepted",
  "message": "Stop command sent to session TXN-90421"
}

# Response 200 OK:

  "message": "Unlock command sent to connector"

4. Identifier Mapping & Granularity

EV charging infrastructure uses a hierarchy of identifiers, and different actions operate at different levels. This is one of the most common questions we get, so here’s how we handle it.

4a. Typical Hierarchy

LevelWhat It RepresentsExampleTypical Actions
Location Physical site (parking garage, service station) LOC-001 Batch queries, location-level filtering
Station / Charger Physical charging unit (the box on the wall or pedestal) STATION-1902 Reset
EVSE Electrical supply point (one station may have 1–4 EVSEs) DE*ABC*E1902*1 Status, start/stop
Connector Physical plug (CCS, Type 2, CHAdeMO) EVSE-8171-2 Unlock, stop transaction

4b. How We Handle This

On our side, the customer always identifies the charger at connector level (e.g. “I’m at connector 8171-2”). Our system maps this to whatever ID your API expects for each action:

  • Reset is at station level → we map 8171-2 to station 1902 and call your reset endpoint with 1902
  • Unlock is at connector level → we use 8171-2 directly
  • Stop depends on your API → we use whichever ID your stop endpoint expects
Key requirement: The identifiers used in remote action endpoints must be clearly mappable to what your station information endpoint returns. For example, if your reset endpoint expects a stationId, that same stationId must appear in the bulk station data so we can link it to the connector-level ID the customer gives us. As long as we can go from a connector ID → to whatever ID each action needs, we’re good. This can be a hierarchy in the station data (location → station → EVSE → connector), or documentation that explains the mapping.

5. Authentication

We support all standard authentication methods. Whatever you already use for your existing APIs works fine — the key is that it’s consistent and clearly communicated.

MethodHow It WorksPrevalence
OAuth2 client credentials We call a token endpoint with client_id + client_secret, receive a time-limited access token, and include it as Authorization: Bearer {token}. We handle token refresh automatically. Most common (~40% of integrations)
API key Static key sent as a header (e.g. Authorization: Bearer {key} or a custom header like X-API-Key). Simplest to implement. Common (~35%)
JWT / M2M token Long-lived machine-to-machine token, typically provided once and valid for months. Sent as Bearer token. Occasional (~15%)
Custom / other Reuse of OCPI token handshake, Azure subscription keys, username/password flows — we can adapt to any mechanism. Rare (~10%)
Bottom line: Use whatever you already have. If starting from scratch, OAuth2 client credentials or a simple API key are both excellent choices.

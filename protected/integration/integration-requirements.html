API Integration Requirements

April 2026
This document provides an overview of the endpoints required for integration with Lemonflow. Information may be scattered across multiple endpoints in your system — this reflects the ideal state. For detailed best practices and examples on specific topics, see the linked deep-dive guides.

Endpoints Required for Integration

Required Recommended
Endpoint Preferred Query Fields Response Structure Voice / Chat Health
Bulk Station Information Initial data import and periodic full synchronisation (1x/day) GET request
/api/charging-stations (or similar)

Pagination: Must support pagination for large datasets (100+ stations) 		Unique IDs (required):
• Location / Station / Connector IDs
• EVSE ID

Charger Info (recommended):
• Connector type (Type 2, CCS, etc.)
• Maximum power (kW)
• Voltage & Amperage (optional)

Location Data (recommended):
• GPS coordinates (latitude, longitude)
• Street address
• City + Postal code
• Country code (2-letter ISO)

Additional Data (recommended):
• Manufacturer & model
• Serial number, firmware version
• Pricing / tariff information
• Fields for analytics purposes
Single Station or Connector Endpoint Real-time status updates during active charging sessions GET or POST request
/api/v1/charging-stations/{station_id} (or similar)

Batch supported (optional): Ideally endpoint supports batch processing for multiple stations, e.g. array of comma-separated station IDs 		Status (required):
• Status of station (AVAILABLE, CHARGING, etc.)

Technical requirements (required):
• Must respond within less than 2 seconds
• Rate limit >1000 rpm
• Ideally same data structure as bulk endpoint
OCPP Endpoint Detailed charging session analytics and diagnostics GET request
/api/v1/stations/{station_id}/ocpp-logs

Time range filtering (recommended):
Allow from-to dates (unix or ISO time)

Max logs filtering (recommended):
Alternative to time range, most recent X logs returned

Pagination: Must support pagination, or very large page sizes (1 day ≈ approx. 5k OCPP entries per charger) 		Data format:
• Array of OCPP message logs with timestamps
• OCPP 1.6 or 2.0.1 currently supported
Remote Actions Endpoint Charger control during support calls POST requests

Reset Charger
/api/v1/chargers/{id}/reset

Stop Charging
/api/v1/connectors/{id}/stop or as alternative /api/v1/sessions/{id}/stop (then /active_sessions endpoint is also required)

Unlock Cable
/api/v1/connectors/{id}/unlock 		Success/failure message
• Response message if success/failure
Ticketing Support ticket creation POST request

Standard endpoint that allows fields to be set — fields to be defined by customer (e.g. description, severity, charger_id, evse_id, phone_number, …) 		Success/failure message
• Response message if success/failure
• Ticket ID / number

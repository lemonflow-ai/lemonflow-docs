1. What We Use OCPP Logs For

OCPP logs are the raw protocol messages exchanged between chargers and the CPMS — heartbeats, status notifications, meter values, error codes, transaction events, and more. We use these for real-time diagnostics during support calls. They let the AI agent understand what the charger is actually doing and provide accurate troubleshooting guidance to the end user.

Key OCPP messages we look for

OCPP MessageWhat It Tells UsExample Use
StatusNotification Current charger/connector state and error codes “Connector 2 is in Faulted state with GroundFailure error”
MeterValues Energy delivery, power readings, SoC “The charger is delivering 22 kW” or “Power dropped to 0 kW 5 minutes ago”
StartTransaction / StopTransaction Session lifecycle “Your session started at 08:12 and is still active”
Heartbeat Connectivity “The charger last communicated 3 minutes ago” vs. “The charger has been offline since yesterday”
BootNotification Charger restarts “The charger restarted at 10:45 — this may have interrupted your session”
DiagnosticsStatusNotification Firmware/diagnostic activity Identifying if maintenance activity is affecting availability
The more complete the logs, the more accurate the AI agent’s troubleshooting. Raw, unfiltered OCPP data is ideal — we handle the parsing and interpretation on our side. Both OCPP 1.6 and OCPP 2.0.1 message formats are supported.

2. Best Practices

Based on what we’ve seen across 15+ CPMS integrations, here are the patterns that work best for OCPP log endpoints:

PracticeDetails
Time-based queries
(not count-based)		 The endpoint should accept from and to timestamps (ISO 8601 with timezone) and return all OCPP messages within that window. This avoids blind spots that can occur with “latest N messages” style endpoints. Supporting at least 24 hours per query is useful; ~72 hours is ideal.
Generous page sizes ~1,000 records per page works well. A typical ~6-hour window produces ~300–500 messages, so this often allows retrieval in a single request. Cursor-based pagination is most reliable (e.g. next_cursor token). Offset/limit also works if there’s a clear “has more” indicator.
Filtering & batch Ideally queryable by EVSE ID, charger ID, and/or location ID. Batch support (up to ~20 IDs per request) is very useful — we often poll the charger a customer is at plus 2–3 nearby chargers for context.
Retention At least 72 hours minimum, ~7 days preferred for incident investigation.
Sort order Newest first (descending) — we’re almost always interested in the most recent activity first.
Raw OCPP data Original OCPP message payloads, as raw and unmodified as possible. Pre-processed or summarised data can work but often loses detail important for edge-case diagnosis.
Response time Under ~2 seconds per page. In practice, most CPMS endpoints respond in 150–400 ms.
Rate limits ~1,000 requests per minute is a common baseline. Standard HTTP 429 with Retry-After header if throttled.
Authentication We support OAuth2, API keys, and custom header-based auth — whatever you already use. See Section 5.

3. Typical API Structure

ParameterRecommendedNotes
HTTP method GET POST with filter body is also common (especially for batch queries)
Endpoint pattern /api/v1/stations/{id}/ocpp-logs Or /chargepoints/{id}/logs, /controllers/{id}/messages — naming varies
Time range from + to (ISO 8601 with timezone) e.g. 2026-04-07T06:00:00Z. Alternative: date_from, date_to or createdAfter, createdBefore
Max time range At least 72 hours per query No hard cap is ideal; ~72h is a comfortable minimum
Pagination Cursor-based (preferred) or offset/limit Include a next_cursor, has_more, or next_page indicator
Page size Up to 1,000 per page Configurable via limit or per_page parameter
Sort order Newest first (descending) Configurable is a nice-to-have, but descending default is fine
Filtering By charger/station ID (required) EVSE ID and connector ID filtering are useful extras. Batch (list of IDs) is a bonus.
Data format Raw OCPP message payloads with timestamps OCPP 1.6 or 2.0.1 — we handle both
Retention 72 hours minimum, 7 days preferred Longer is better for incident investigation
What about WebSocket / streaming / Kafka? We can work with real-time event streams, but a REST endpoint with time-based queries is operationally simpler for our use case. During a support call, we typically need to look back at the last few hours of OCPP history — not just live events. A GET endpoint gives us that historical window reliably, without needing persistent connections or handling reconnection gaps. If you only have a streaming interface, we can integrate it, but it requires an initial historical backfill which adds complexity on both sides.

4c. Batch Query (Optional)

If supported, batch queries allow us to efficiently fetch logs for a charger and its neighbours in a single request:

POST /api/v1/ocpp-logs/query
Authorization: Bearer {token}
Content-Type: application/json

{
  "stationIds": ["STATION-1902", "STATION-1903", "STATION-1904"],
  "from": "2026-04-07T06:00:00Z",
  "to": "2026-04-07T12:00:00Z",
  "limit": 1000
}
Why batch? During a support call, if a customer reports an issue at one charger, we often check the neighbouring chargers too — it helps distinguish between a single-charger fault and a site-wide issue (e.g. power supply problem). Batch support makes this efficient.

5. Authentication

We support all standard authentication methods. Whatever you already use for your existing APIs works fine — the key is that it’s consistent and clearly communicated.

MethodHow It WorksPrevalence
OAuth2 client credentials We call a token endpoint with client_id + client_secret, receive a time-limited access token, and include it as Authorization: Bearer {token}. We handle token refresh automatically. Most common (~40% of integrations)
API key Static key sent as a header (e.g. Authorization: Bearer {key} or a custom header like X-API-Key). Simplest to implement. Common (~35%)
JWT / M2M token Long-lived machine-to-machine token, typically provided once and valid for months. Sent as Bearer token. Occasional (~15%)
Custom / other Reuse of OCPI token handshake, Azure subscription keys, username/password flows — we can adapt to any mechanism. Rare (~10%)
Bottom line: Use whatever you already have. If starting from scratch, OAuth2 client credentials or a simple API key are both excellent choices.
